# OpenClaw 科研证据链插件-- by David

工作总结：插件四次重大版本更新，提升插件性能和稳定性，完成多达30次case study，编写最后的技术细节

- 当前仓库：`https://github.com/CrystalDavid/openclaw-evidence-public.git`


---

## 1. 项目概况

实现功能：
1. 区分 Agent 的推理与真实材料中的直接证据。
2. 验证引用的论文是否真实存在、身份是否正确。
3. 真正打开上传 PDF 和外部论文全文，检查 Claim 是否被原文支持。
4. 对每条 Claim 给出来源、页码、证据片段、支持度和判定理由。
5. 保留错误、证据不足和下载阻塞情况，不允许为了报告好看而替换或删除失败证据。

当前系统把“发现候选文献”和“验证证据真实性”明确分开：

- **OpenClaw / DeepSeek Agent**：阅读源论文、检索候选文献、提出三个研究 Idea、拆分 Claim。
- **evidence-tracker 插件**：管理论文身份、下载真实全文、定位证据、独立判定并生成报告。

## 2. 当前系统架构

### 2.1 QQBot 与 OpenClaw Agent

用户通过 QQBot 上传 PDF，并使用简单提示词，例如：

```text
用证据链追踪插件帮我分析这个 PDF
```

OpenClaw Agent 负责：

1. 找到本次 QQBot 消息对应的原始上传 PDF。
2. 启动新的 evidence session。
3. 阅读 PDF 的研究目标、方法、实验、局限性和相关工作。
4. 使用学术检索工具寻找外部英文论文。
5. 基于源论文、外部文献和模型推理生成三个研究 Idea。
6. 把 Idea 拆成可以单独核验的原子 Claim。
7. 逐条调用 evidence-tracker 工具。

当前发现阶段优先使用专门的学术检索工具，不使用通用 Web Search 冒充学术证据：

- evidence-tracker 的 `search_literature`
- `paper-search` 插件
- `scientify` 的 arXiv / OpenAlex 检索能力

这些发现器用于增加候选论文覆盖率；最终判定仍必须由 evidence-tracker 独立完成。

### 2.2 evidence-tracker 插件

当前插件公开六个工具：

1. `start_evidence_tracking`
2. `search_literature`
3. `record_citation`
4. `verify_evidence_chain`
5. `generate_evidence_report`
6. `get_session_status`

它们分别负责启动会话、发现文献、记录 Claim、核验证据、生成报告和查看状态。

## 3. 四次重大更新与差异

说明：v2、v3、v4 使用指定 Git 快照比较。一个快照可能包含其与上一版本之间的多个中间提交，因此下面描述的是“该版本相对上一版本最终新增的能力”，而不只是该命名提交单独修改的几行代码。

### v1：原始插件压缩包

版本来源：

```text
C:\Users\David\Downloads\openclaw-evidence-tracker (1).zip
```

#### v1 的实现

- 原始主版本注册五个工具：
  - `start_evidence_tracking`
  - `record_citation`
  - `verify_evidence_chain`
  - `generate_evidence_report`
  - `get_session_status`
- 主要通过 DOI、arXiv、标题等元数据确认论文存在。
- Claim 支持关系主要依据摘要和 LLM 判断。
- 输出 GREEN / YELLOW / RED 报告。
- 压缩包中还保留了 `index-old-auto.ts`、`index-v2.ts` 等不同实验路线。

#### v1 的主要问题

- `search_literature` 没有在主版本中公开给 Agent。
- 不会真正打开用户上传 PDF 验证页码和引句。
- 外部论文通常只验证摘要，不下载并核验全文。
- Agent 写入的 Context 对判定影响过大。
- 没有明确的 `source_type`，容易混淆源论文证据和外部论文证据。
- 没有论文身份注册表、跨数据库去重和下载缓存身份检查。
- API 或全文不可用时容易产生不稳定的 YELLOW / RED。

v1 是“证据记录与元数据验证原型”，还不是严格全文审计系统。

### v2：`d7d6743 Generate evidence reports as PDF by default`

版本来源：

```text
https://github.com/CrystalDavid/openclaw.git
commit d7d6743
```

#### 相比 v1 的变化

- 插件正式接入本地 OpenClaw 与 QQBot 工具链。
- 修复了早期 `record_citation`、`verify_evidence_chain` 等工具在实际会话中不可见或调用不稳定的问题。
- 增加 Markdown 转 HTML / PDF 的能力，`generate_evidence_report` 可以直接输出 PDF。
- 增加更明确的 GREEN / YELLOW / RED 判定提示。
- 增加确定性降级规则，用于纠正部分明显过度声明。
- 报告开始包含 Evidence Table、支持度和错误类型。

#### v2 修复的问题

- 解决了“插件存在，但 Agent 说工具不存在”的主要注册与调用问题。
- 解决了无法直接生成可交付 PDF 报告的问题。
- 对部分明显错引、强证明语言和新 Idea 过度判绿增加了兜底。

#### v2 的局限

- 核验核心仍主要依赖元数据、摘要和 Agent Context。
- PDF 内证据仍可能因为 Context 中出现页码、章节或引句而直接判绿，并未真实打开 PDF 审核。
- 外部论文没有稳定的全文下载与逐页核验。
- 一些确定性规则包含 PromptIR、频域等特定研究场景，不适合通用论文。
- PDF 排版质量不稳定。

v2 的核心贡献是“打通真实交互和 PDF 交付”，但证据真实性仍未彻底解决。

### v3：`73944d6 Back up OpenClaw state and metadata layout`

版本来源：

```text
https://github.com/CrystalDavid/openclaw.git
commit 73944d6
```

#### 相比 v2 的变化

- 报告输出从 PDF 优先调整为高质量中文 Markdown。
- `search_literature` 正式成为第六个公开工具。
- 文献检索扩展到 Semantic Scholar、OpenAlex、CrossRef、Europe PMC、DBLP，并增强 arXiv fallback。
- 修复 APA 格式、带 venue 格式和 arXiv 引用的标题提取问题。
- 增加 DOI、arXiv、标题、作者年份的多源查找和标题兼容检查。
- 增加重试与 fallback，降低单一 API 失败带来的误判。
- 增加至少 15 条 Citation、至少 6 条外部文献的质量门槛。
- 报告改为两个清晰部分，并完善中文标题、彩色状态、元信息列表和 Markdown 排版。
- 输出文件名改为 `<研究方向>_evidence.md`。

#### v3 修复的问题

- 修复外部英文文献数量太少、只依赖单一检索源的问题。
- 修复 arXiv 长期超时、某个 API 失败就直接放弃的问题。
- 修复 APA 标题提取错误导致真实论文被判不存在的问题。
- 修复固定输出 `example2.md`、英文报告、标题重复和元信息挤在一行的问题。
- 修复 PDF 输出质量不稳定的问题，转向 Markdown-only。

#### v3 的局限

- PDF 内证据仍主要依靠 Agent 提供的页码、章节、引句和 Context 判断。
- 外部文献虽然多源确认身份，但支持关系主要仍基于摘要。
- 没有完整的论文 canonical registry 和严格的跨库去重。
- 下载失败、论文不存在和内容不支持之间仍可能混淆。
- Agent 仍可能通过替换 RED / BLOCKED 引用让报告看起来更好。

v3 的核心贡献是“多源学术检索、Markdown 报告和质量门槛”。

### v4：`bace4c0 Fix decimal percentage evidence matching`

版本来源：

```text
https://github.com/CrystalDavid/openclaw.git
commit bace4c0
```

v4 快照累计包含从 v3 到当前版本的审计强化，包括论文注册表、全文核验、防替换作弊和数值匹配修复。

#### 相比 v3 的变化

1. **真正打开上传 PDF 审核**
   - `start_evidence_tracking` 必须接收并解析真实 `source_pdf_path`。
   - PDF 内证据不再因 Agent 写了页码或长 Context 就直接判 GREEN。

2. **外部论文真实全文核验**
   - 多源确认论文身份后，继续寻找并下载真实 PDF。
   - 从实际全文逐页检索证据，不再只看摘要。

3. **多源下载与身份冲突处理**
   - 增加 arXiv、OpenAlex、Semantic Scholar、Unpaywall、CrossRef、Europe PMC / PMC、DataCite、HAL 和会议仓库等下载路径。
   - 下载后检查首页标题、DOI 和 arXiv。
   - 错误 PDF 会被删除并标记 `IDENTITY_CONFLICT`。

4. **论文注册表与严格去重**
   - 增加统一 `literature_registry.jsonl`。
   - 通过 canonical ID 合并同一论文。
   - DOI / arXiv 等强标识符冲突时禁止污染合并。

5. **明确来源类型**
   - `record_citation` 强制使用 `uploaded_pdf` 或 `external_literature`。
   - 来源身份不一致时自动纠正为外部论文并重新下载。

6. **原子 Claim 与防作弊机制**
   - 一条 Claim 只能包含一个结论并引用一篇论文。
   - 自动拆分组合 Claim。
   - 禁止用 `replace_citation_id` 更换 Claim 或论文。
   - RED / BLOCKED 必须原样保留，不能通过换证据洗白。

7. **混合全文检索**
   - 使用 BM25、轻量哈希向量、数值匹配、页码/章节提示和 LLM 查询扩展召回片段。
   - 再使用 LLM 做 `ENTAILMENT / PARTIAL / CONTRADICTION / NOT_FOUND` 判断。

8. **BLOCKED 独立统计**
   - 下载或工程问题不再伪装成 YELLOW / RED。
   - 所有下载源耗尽后保留 BLOCKED，并继续生成透明报告。

9. **session 缓存治理**
   - 每个新 session 重新执行外部检索。
   - 报告完成后和新 session 开始前清理全文缓存。
   - 保留 canonical metadata 和下载历史，但重新下载并重新做身份校验。

10. **小数百分比匹配修复**
    - 修复 `24.06%` 被错误识别出 `06%`，导致精确数值证据被误判缺失的问题。
    - 数值现在按完整 token 标准化比较，支持小数、空格百分号和千位分隔符。

#### v4 修复的问题

- 修复“插件根本没有打开 PDF，却说证据已验证”的根本问题。
- 修复“外部论文只验证摘要，无法证明具体 Claim”的根本问题。
- 修复下载到错误论文后污染后续任务的问题。
- 修复相同论文跨数据库重复搜索、重复下载、重复计数的问题。
- 修复多个来源身份冲突时错误合并的问题。
- 修复组合 Claim 无法准确判定的问题。
- 修复 Agent 为减少 RED / BLOCKED 而替换引用的作弊路径。
- 修复 API / 下载失败被错误混入科学判定的问题。
- 修复小数百分比被截断导致的假 YELLOW。

v4 的核心贡献是把项目从“多源摘要验证器”升级为“带论文身份管理、真实全文下载、逐页证据定位和透明工程状态的证据审计系统”。
